1. THE VISION
================================================================================

Build an end-to-end autonomous drone system for one-click 3D Gaussian splat
reconstruction. The user places an object, the drone autonomously captures it
from all angles, and a pipeline outputs a polished 3D model. No manual flying.
No manual photogrammetry. Press a button, get a 3D model.

COMPLETE PIPELINE:
  1. User places object on surface
  2. Drone takes off and flies over object to measure height (ToF/VPS differential altimetry)
  3. Indoor positioning system tracks drone location in real-time (no GPS)
  4. Drone executes multi-altitude dome capture pattern around object
  5. Real-time object recognition identifies and tracks target
  6. Gaussian splatting pipeline generates 3D reconstruction
  7. User receives polished 3D model

3. WHAT WAS BUILT: PYTHON WAYPOINT MISSION GENERATOR
================================================================================

3.1 WPML REVERSE ENGINEERING
-----------------------------
DJI's waypoint missions use the WPML format, which is essentially a KMZ file
(ZIP archive) containing XML files. This format was reverse-engineered from DJI's
Cloud API documentation and practical experimentation.

WPML File Structure:
  - KMZ file (ZIP format) containing template.kml and waylines.wpml
  - template.kml defines mission metadata, drone model, payload settings
  - waylines.wpml contains the actual waypoint coordinates and actions
  - Supports waypoint types: fly-to-point, curved flight, hover
  - Actions at waypoints: take photo, start/stop recording, gimbal rotation

3.2 DOME CAPTURE PATTERN
-------------------------
The Python generator creates multi-altitude orbital dome capture patterns
optimized for 3D reconstruction:

  Parameter          Value                        Purpose
  ----------------   --------------------------   -------------------------
  Total waypoints    58                           Full dome coverage
  Altitude levels    4 (0.9m, 1.7m, 2.7m, 3.3m)  Multi-elevation capture
  Elevation bands    25°, 50°, 75°                Hemisphere coverage
  Points per band    10, 8, 5 (lower = more)      Denser at lower angles
  Dome radius        3.0m (adjustable)            Distance from object
  Flight speed       <1 m/s precision             Smooth capture
  POI tracking       360° continuous              Camera always faces object
  Gimbal angles      -25°, -50°, -75°             Progressive downward tilt

3.3 KEY TECHNICAL FEATURES
--------------------------
  - Continuous POI tracking: Drone heading always faces center object at every waypoint
  - Smooth curved flight paths: Uses WPML curved turn mode for cinematographic quality
  - Gimbal automation: Pitch calculated per waypoint based on elevation angle to object center
  - Video recording automation: Start/stop recording embedded in waypoint actions
  - GPS coordinate generation: Converts dome geometry into lat/lon via distance-bearing calcs
  - Configurable parameters: Object height, dome radius, bands, points all adjustable

3.4 SD CARD DEPLOYMENT WORKAROUND
----------------------------------
With the RC 2 controller (no MSDK access), a workaround was discovered:

  1. Create a dummy waypoint mission in DJI Fly (requires outdoor GPS lock)
  2. Access the drone's SD card via the RC 2 file manager
  3. Navigate to the waypoint folder (random hex-named directories)
  4. Replace the .kmz file with the custom-generated dome capture mission
  5. Load mission in DJI Fly and execute

  >>> This method was validated but has limitations: GPS waypoints don't work
  >>> indoors, and the workaround is fragile.

4. INDOOR POSITIONING RESEARCH
================================================================================

GPS-based waypoint missions do NOT work indoors. This is a hard limit in DJI's
MSDK - the WaypointMission class requires GPS lock to initialize. All indoor
navigation must use the Virtual Stick API with an external positioning system.

4.1 THE CORE PROBLEM
---------------------
The DJI Mini 4 Pro's Vision Positioning System (VPS) handles hover stability
effectively - the drone can hold position indoors. However, VPS does NOT provide
absolute room coordinates. It only maintains relative position from its current
hover point. Over time, drift accumulates, affecting multi-orbit coverage patterns.

  >>> FUNDAMENTAL CONFLICT: The main gimbal camera is the only camera accessible
  >>> via MSDK. If you point it at positioning markers for navigation, you can't
  >>> simultaneously point it at the object for capture.

4.2 POSITIONING ALTERNATIVES EVALUATED
---------------------------------------

OPTION 1: MARVELMIND INDOOR GPS (RECOMMENDED)
  - Ultrasonic beacon-based positioning, ±2cm accuracy
  - Proven working with DJI Mini 4 Pro + RC-N2 setup
  - Camera-independent — no conflict with capture gimbal
  - Requires 4+ stationary beacons + 1 mobile beacon on drone
  - Cost: $500-800 for starter kit + software license
  - Android app streams position via UDP, controls drone via Virtual Stick

OPTION 2: ARUCO MARKERS
  - Printed fiducial markers detected via OpenCV
  - Free to produce, well-documented library support
  - Camera must see them — conflicts with gimbal pointing at subject
  - Accuracy degrades beyond 3-4m, affected by lighting
  - Possible interleaved workaround:
    gimbal down (detect) → fly segment → gimbal to subject (capture) → repeat
  - Works for still photos only, not continuous video

OPTION 3: APRILTAGS
  - Better-engineered alternative to ArUco
  - Lower false detection rates, more robust at oblique angles
  - Same fundamental camera conflict limitation

OPTION 4: UWB (ULTRA-WIDEBAND) SYSTEMS (e.g., Pozyx)
  - Camera-independent positioning
  - Less precise than Marvelmind for this application
  - Requires anchor infrastructure around the room

OPTION 5: VISUAL SLAM ON MAIN CAMERA
  - Free, no additional hardware needed
  - Complex to implement, conflicts with capture (camera shared)
  - Heavy computation — best offloaded to nearby laptop

OPTION 6: CEILING-MOUNTED CAMERA TRACKING
  - External camera(s) mounted above track the drone from outside
  - Camera-independent on the drone side
  - Leverages existing Python/CV skills
  - Requires room infrastructure setup

OPTION 7: COLORED/PATTERNED FLOOR MATS
  - Large printed mat with unique gradient or pattern
  - CV determines position by recognizing visible section
  - Same camera conflict as marker-based approaches

4.3 KEY TECHNICAL INSIGHT
-------------------------
For the orbital capture use case, camera-independent solutions (Marvelmind, UWB,
ceiling cameras) are strongly preferred because they allow the gimbal to focus
entirely on the capture subject while the drone navigates autonomously.

5.2 MSDK INDOOR FLIGHT: VIRTUAL STICK API
------------------------------------------
Since WaypointMission requires GPS, all indoor flight uses Virtual Stick Control Mode:

  - Send FlightControlData: pitch, roll, yaw, and vertical throttle
  - Implement closed-loop feedback using external position data
  - PID controller needed to handle latency in the control chain
  - Control chain latency: Drone → Remote → Phone → Server → Phone → Remote → Drone

5.3 VIDEO STREAMING FOR EXTERNAL PROCESSING
--------------------------------------------
MSDK provides access to the live video stream via the VideoFeeder API:

  - Raw YUV or H.264 frames accessible on the Android device
  - Can stream via TCP/UDP/WebRTC to external compute (laptop/server)
  - For lightweight drones like Mini 4 Pro, heavy processing (SLAM, object
    detection) is better offloaded to nearby laptop
  - Local processing possible with OpenCV/TFLite for low-latency tasks

6. OBJECT HEIGHT MEASUREMENT CONCEPT
================================================================================

Before executing the dome capture pattern, the drone needs to know the object's
height to calculate optimal orbit altitudes and gimbal angles.

6.1 DIFFERENTIAL ALTIMETRY APPROACH
------------------------------------
  1. Drone hovers over the floor next to the object → measures distance to floor (h_floor)
  2. Drone moves over the object → measures distance to object top (h_object)
  3. Object height = h_floor - h_object (at same drone altitude)

  ToF Formula:      h = c·t/2 (speed of light × photon round-trip time / 2)
  Target Accuracy:  Sub-decimeter in GPS-denied environments

6.2 FEASIBILITY ASSESSMENT
---------------------------
  - VPS sensors perform these calculations internally for hover stability
  - RC 2 doesn't expose raw sensor data; MSDK with RC-N3 may provide altitude readings
  - Practical workaround: Use displayed altitude readings for manual-assisted measurement
  - Concept is sound for single-altitude orbits at close range
  - Multi-altitude missions need position correction to counter VPS drift over time

7. MSDK TECHNICAL ANALYSIS (FROM RIIS LLC RESEARCH)
================================================================================

Based on analysis of RIIS LLC's deep dives into DJI Mobile SDK, these are the
key findings for indoor autonomous flight:

7.1 THREE CORE QUESTIONS ANSWERED
----------------------------------

Q1: Is vision-based positioning reliable for indoor flight paths?
  - VPS is stable for hovering and short-distance maneuvers
  - Drift is inevitable over long paths — no true indoor GPS
  - Closed-loop feedback system required using video feed or external sensors
  - Virtual Stick Control Mode provides FlightControlData (pitch, roll, yaw, throttle)

Q2: Can camera feed stream to external compute in real-time?
  - Yes — core MSDK capability via VideoFeeder API
  - Socket streaming (TCP/UDP/WebRTC) from mobile app to external server
  - For lighter drones, offload heavy computation (SLAM, detection) to nearby laptop
  - On-device processing possible with OpenCV or TFLite for low-latency needs

Q3: Hard limitations for indoor autonomous navigation?
  - NO WAYPOINT MISSIONS INDOORS: WaypointMission class requires GPS lock — hard
    limit. Must use Virtual Stick API exclusively.
  - NETWORK LATENCY: Full round-trip chain (drone→remote→phone→server→back) causes
    oscillations. PID tuning critical.
  - MSDK V4 vs V5 INCOMPATIBILITY: V5 has completely different architecture. V5 is
    Android-only with different streaming APIs. Must match SDK version to hardware.

8. DRIFT & POSITION CORRECTION STRATEGIES
================================================================================

VPS drift accumulation is a key challenge for multi-orbit dome capture missions.
Several strategies were discussed:

  - Return to a known position between altitude layers to reset drift reference
  - Use visual markers on the ground for continuous position correction
  - Accept some imperfection and rely on photogrammetry software (Meshroom,
    RealityCapture) to handle slight overlap inconsistencies — these tools are
    surprisingly forgiving
  - For single-altitude orbits at close range, drift is minimal and acceptable
  - For full dome coverage (multiple altitudes), active correction becomes important
  - Marvelmind or UWB provides continuous absolute position feedback independent of drift
